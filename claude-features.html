#1 압도적 1위 가장 많이 쓰는 기능
HOT 🎯

Goal 실행 — "이거 해줘"

Claude Code의 핵심. 슬래시 명령 없이 목표를 자연어로 말하면 파일을 읽고, 코드를 수정하고, git까지 알아서 처리합니다. 대부분의 사용자가 하루 대부분을 이 방식으로 씁니다.

  • → "이 버그 고쳐줘 / 이 에러 원인 찾아줘"
  • → "로그인 기능 JWT + 소셜 로그인 포함해서 만들어줘"
  • → "테스트 코드 작성해줘 / 리팩토링해줘"
"결제 페이지에서 PayPal 버튼이 안 눌려, 고쳐줘"
"이 함수 너무 복잡해, 단순하게 리팩토링해줘"
⚠ 단점

목표가 불명확하거나 범위가 너무 크면 Claude가 계속 파일을 읽고 수정하면서 무한루프에 빠져 토큰을 대량 소비합니다. "전체 리팩토링해줘" 같은 모호한 지시가 특히 위험합니다. 중간에 방향을 바꾸면 앞서 수정한 내용과 충돌이 나기도 합니다.

✓ 해결책

범위를 명확하게 한정하세요. "이 파일의 이 함수만", "이 3개 파일만 건드려줘" 식으로 구체적일수록 좋습니다. 큰 작업은 단계를 나눠서 하나씩 확인하면서 진행하고, Plan Mode를 먼저 써서 접근 방식을 확인한 뒤 승인하세요.

자연어 명령 파일 자동 탐색 멀티 파일 편집 git 자동화
#2 딥 서치 & 웹 검색
HOT 🔍

웹 검색 (WebSearch)

모르는 에러, 최신 API 문서, 라이브러리 호환성을 실시간으로 검색합니다. 학습 데이터가 아닌 지금 웹의 최신 정보를 가져옵니다.

  • → 에러 메시지 원인 검색
  • → 최신 npm 패키지 버전 확인
  • → 공식 문서 최신 API 조회
⚠ 단점

검색 결과가 오래됐거나 신뢰도 낮은 사이트 내용을 그대로 쓸 수 있습니다. 여러 번 검색하면 컨텍스트를 많이 소모해 속도가 느려집니다.

✓ 해결책

공식 문서 URL을 직접 알려주면 더 정확합니다. "검색 말고 이 URL 읽어줘" 방식으로 WebFetch를 쓰면 신뢰도가 훨씬 높아집니다.

🌐

웹 크롤링 (WebFetch)

특정 URL의 내용을 직접 읽어옵니다. 공식 문서, GitHub README, 블로그를 크롤링해서 코드에 반영합니다.

  • → 공식 문서 페이지 내용 읽기
  • → GitHub 이슈/PR 내용 파악
  • → 특정 페이지 내용 기반 코드 작성
⚠ 단점

JavaScript로 렌더링되는 SPA 페이지는 내용을 제대로 못 읽습니다. 큰 페이지는 중간에서 잘립니다. 로그인이 필요한 페이지는 접근 불가합니다.

✓ 해결책

섹션이 있는 URL (앵커 포함)을 주거나, 필요한 내용을 직접 복사해서 대화에 붙여넣으면 더 정확합니다.

#3 코드베이스 탐색
🗂

파일 & 코드 탐색

Glob 패턴 + 정규식 검색으로 대규모 프로젝트에서 원하는 파일이나 함수를 순식간에 찾습니다.

  • → "이 함수 어디서 쓰이는지 찾아줘"
  • → "결제 관련 파일들 다 보여줘"
⚠ 단점

큰 프로젝트에서 너무 광범위하게 탐색하면 수십 개 파일을 읽어서 컨텍스트 윈도우를 빠르게 채웁니다. 컨텍스트가 꽉 차면 이전 내용이 압축돼 기억이 흐려집니다.

✓ 해결책

탐색 범위를 좁혀 지시하세요. "src/payment 폴더만", "이 파일들만 봐줘" 식으로 디렉토리를 한정합니다. Explore 에이전트를 써서 메인 컨텍스트를 보호하는 것도 방법입니다.

Glob Grep 심볼 탐색
🔎

Explore 에이전트

복잡한 탐색을 전담 서브 에이전트에게 위임해서 메인 컨텍스트를 보호합니다. 처음 보는 레포 구조 파악에 유용합니다.

  • → 아키텍처 빠르게 파악
  • → 의존성 관계 추적
⚠ 단점

파일 전체가 아닌 excerpt(발췌)만 읽기 때문에 파일 끝부분 내용을 놓칩니다. 코드 리뷰나 깊은 분석에는 부적합합니다.

✓ 해결책

위치 파악(어디에 뭐가 있는지)에만 쓰고, 실제 내용 분석은 Claude가 직접 Read 하도록 하세요. "이 파일 전체 읽어줘"라고 명시하면 됩니다.

#4 Git & PR 작업
📦

Git 커밋 & PR

변경 사항을 파악하고 컨벤션에 맞는 커밋 메시지를 자동 생성, PR 설명까지 작성합니다.

  • → "커밋해줘" → 변경 분석 후 메시지 생성
  • → "PR 만들어줘" → 제목/본문 자동 작성
⚠ 단점

.env, API 키, 비밀번호가 담긴 파일이 실수로 커밋될 수 있습니다. git add -A를 쓰면 의도치 않은 파일까지 포함됩니다. force push를 요청하면 팀원 작업이 날아갈 수 있습니다.

✓ 해결책

.gitignore를 꼼꼼히 관리하고, "커밋 전에 뭘 추가할지 먼저 보여줘"라고 확인 단계를 거치세요. Claude는 기본적으로 민감 파일을 경고하지만, 명시적으로 물어보는 게 안전합니다.

👀

PR 리뷰 (/review)

PR 전체를 검토하고 문제점, 개선점을 정리합니다. 머지 전 마지막 검토에 사용합니다.

  • → PR 머지 전 마지막 검토
  • → 팀원 PR 빠르게 파악
/review
⚠ 단점

변경 파일이 많으면 모든 파일을 꼼꼼히 보지 않고 표면적인 리뷰에 그칩니다. 비즈니스 로직의 맥락을 모르면 중요한 버그를 놓칠 수 있습니다.

✓ 해결책

배경 맥락을 함께 알려주세요. "이 PR은 결제 실패 재시도 로직 추가야, 특히 타임아웃 케이스 집중해서 봐줘" 식으로 초점을 주면 리뷰 질이 올라갑니다.

#5 코드 리뷰
⚡

/code-review

현재 브랜치 변경 사항을 리뷰합니다. low~high 레벨로 깊이 조절 가능합니다.

/code-review
/code-review high
⚠ 단점

low/medium 레벨은 복잡한 버그를 놓칩니다. 컨텍스트 없이 리뷰하면 false positive(없는 문제 지적)도 나옵니다.

✓ 해결책

중요한 코드는 high 이상으로 올리고, 변경 의도를 먼저 설명한 뒤 리뷰 요청하세요.

ULTRA 🤖

/code-review ultra

멀티 에이전트가 클라우드에서 동시에 돌며 딥 리뷰. PR 인라인 코멘트 자동 작성 가능.

/code-review ultra
/code-review ultra --fix
⚠ 단점

클라우드 실행이라 비용이 발생하고 시간이 오래 걸립니다. 작은 변경에 ultra를 쓰면 낭비입니다. --fix는 리뷰 지적 사항을 자동으로 수정하는데, 원치 않는 변경이 생길 수 있습니다.

✓ 해결책

큰 PR이나 배포 직전에만 ultra를 씁니다. --fix는 수정 전 diff를 꼭 확인하세요. 소규모 변경엔 그냥 /code-review high로 충분합니다.

✂

/simplify

변경된 코드를 더 단순하게 리팩토링합니다. 버그 탐지가 아닌 품질 전용입니다.

/simplify
⚠ 단점

단순화 과정에서 과도한 추상화를 도입하거나 기존 코드 스타일과 맞지 않는 패턴을 쓸 수 있습니다. 팀 컨벤션을 모르면 오히려 복잡해집니다.

✓ 해결책

적용 전 반드시 diff를 검토하고, "우리 팀은 이 패턴 안 씀" 같은 제약을 먼저 알려주세요. CLAUDE.md에 코딩 컨벤션을 적어두면 자동으로 따릅니다.

#6 앱 실행 & 검증
▶

/run — 앱 직접 실행

코드 변경 후 실제 앱을 실행해서 동작을 확인합니다. 서버, CLI, 브라우저 앱 모두 지원합니다.

/run
⚠ 단점

브라우저 UI 변경은 직접 눈으로 확인이 어렵고, 앱 실행에 필요한 환경 변수나 DB 연결이 없으면 실행 자체가 실패합니다. 실행 결과를 잘못 해석해서 "잘 됨"으로 보고하는 경우도 있습니다.

✓ 해결책

환경 변수 세팅을 미리 확인하세요. UI 검증은 스크린샷이나 직접 브라우저에서 확인하는 것이 가장 확실합니다. Claude가 "잘 됩니다"라고 해도 직접 눌러보는 습관을 들이세요.

✅

/verify — 동작 검증

코드 변경이 실제로 의도한 대로 동작하는지 앱을 띄워서 직접 확인합니다.

/verify
⚠ 단점

타입 체크와 테스트 통과를 "기능이 정상"으로 오해할 수 있습니다. 골든 패스만 확인하고 엣지 케이스는 놓칩니다.

✓ 해결책

"엣지 케이스도 테스트해줘 — 빈 입력, 네트워크 실패, 권한 없는 경우" 식으로 명시적으로 요청하세요.

#7 보안 리뷰
🔒

/security-review — 보안 리뷰

현재 브랜치 변경 사항에서 SQL 인젝션, XSS, 인증 문제, 민감 정보 노출 등 OWASP Top 10을 기준으로 취약점을 자동 탐지합니다.

  • → 사용자 입력 처리 코드 점검
  • → 인증/인가 로직 검토
  • → API 키 노출 여부 확인
/security-review
⚠ 단점

오탐(false positive)이 있습니다. 없는 취약점을 있다고 할 수 있고, 반대로 복잡한 비즈니스 로직에 숨은 취약점은 놓칩니다. 전문 보안 감사(penetration testing)를 대체하지는 못합니다. 변경 파일만 보기 때문에 기존 코드와의 상호작용에서 오는 취약점은 잡기 어렵습니다.

✓ 해결책

1차 필터로 쓰고 중요 시스템은 전문 보안 도구(SAST, Snyk 등)와 병행하세요. 오탐 지적에는 "왜 이게 취약점인지 설명해줘"라고 물어보면 실제 위험인지 판단할 수 있습니다.

SQL 인젝션 XSS 인증 취약점 OWASP Top 10
#8 자동화 & 스케줄링
NEW ⏰

/schedule — 자동 루틴

자연어로 스케줄을 설정하면 Claude가 크론 잡을 만들어 원격 에이전트를 자동 실행합니다.

/schedule "매일 오전 9시 PR 리뷰"
⚠ 단점

잘못 설정하면 계속 실행되면서 비용이 누적됩니다. 스케줄 에이전트가 권한 없는 작업(push, 외부 API 호출 등)을 시도하다 오류 루프에 빠질 수 있습니다. 실행 결과를 모르면 문제가 생겨도 늦게 알아챕니다.

✓ 해결책

먼저 1회 실행 테스트를 하고 스케줄을 설정하세요. 실행 후 알림(슬랙, 이메일)을 함께 설정해서 결과를 모니터링하고, 불필요한 스케줄은 /schedule list로 확인 후 삭제하세요.

🔁

/loop — 반복 실행

정해진 간격으로 같은 작업을 반복합니다. CI 폴링, 배포 상태 확인, 반복 테스트에 씁니다.

/loop 5m /verify
/loop # 자기 페이스 조절
⚠ 단점

컨텍스트가 압축되면 이전 상태를 잊어서 "다 끝났는데 계속 같은 작업을 반복"하는 상황이 생깁니다. 종료 조건이 불명확하면 영원히 돌 수 있습니다. 300초 이상 sleep하면 캐시가 날아가 비용이 늘어납니다.

✓ 해결책

명확한 종료 조건을 지정하세요. "CI가 green이 되면 멈춰줘" 같은 조건부 루프가 좋습니다. 대기 간격은 270초 이하(캐시 유지)나 1200초 이상(비용 절감)으로 설정하세요.

#9 멀티 에이전트 & 계획 모드
🤖

병렬 에이전트

독립적인 작업을 동시에 여러 에이전트가 처리합니다. 대형 리팩토링, 멀티 파일 분석에서 시간을 절약합니다.

  • → 여러 파일 동시 분석
  • → 리서치 + 구현 병렬 실행
⚠ 단점

서로 다른 에이전트가 같은 파일을 동시에 수정하면 충돌이 납니다. 에이전트 결과를 통합할 때 일관성이 깨질 수 있습니다. 각 에이전트가 전체 컨텍스트를 공유하지 못해 전제가 달라질 수 있습니다.

✓ 해결책

진짜 독립적인 작업만 병렬화하세요. 같은 파일을 건드리는 작업은 순서대로 처리해야 합니다. "A 에이전트는 프론트, B 에이전트는 백엔드" 식으로 담당 영역을 명확히 나누세요.

📋

계획 모드 (Plan)

코드를 짜기 전에 구현 전략을 설계합니다. 트레이드오프 분석 후 사용자 승인을 받고 실행합니다.

  • → 큰 기능 추가 전 설계 검토
  • → 리스크 있는 작업 사전 확인
⚠ 단점

계획 단계에서 너무 보수적이거나 과도하게 세밀한 계획을 세워서 실제 구현이 계획과 달라지는 경우가 많습니다. 계획을 세우는 데만 시간이 많이 걸려 간단한 작업에도 오버엔지니어링이 됩니다.

✓ 해결책

간단한 작업엔 Plan Mode 불필요합니다. "이 기능 어떻게 구현할지 먼저 설명해줘, 코드는 아직 짜지 마" 식의 자연어 확인으로 충분한 경우가 많습니다.

#10 설정 & 메모리
📄

/init

CLAUDE.md를 자동 생성합니다. 코드베이스를 분석해서 구조와 규칙을 문서화합니다.

/init
⚠ 단점

자동 생성된 CLAUDE.md가 부정확하거나 중요한 컨벤션을 빠뜨릴 수 있습니다.

✓ 해결책

생성 후 반드시 수동으로 검토하고, 팀 컨벤션과 금지 패턴을 직접 추가하세요.

⚙

update-config
